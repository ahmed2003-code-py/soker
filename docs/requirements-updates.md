# تحديثات المتطلبات — Soker ERP

> إضافات مقترحة على المتطلبات الرئيسية، متفق عليها مع المالكين.
> تُقرأ مع `docs/ERP-Build-Plan.md` و `CLAUDE.md` (المخ المشترك).
> كل بند موسوم بحالته الحالية: **✅ منفّذ** / **🟡 جزئي** / **🆕 جديد (لم يُبنَ بعد)**.

---

## 1) الحساب المؤقت (Temporary Customer) — 🆕 جديد

الوضع الحالي: الفاتورة تدعم **عميل زائر (walk-in)** عبر `guestName` فقط — اسم للطباعة **بدون حساب ولا رصيد**. المطلوب هنا أقوى: حساب مؤقت **له رصيد يُتابَع**.

**المطلوب:**
- إمكانية إنشاء **عميل مؤقت** عند إنشاء فاتورة كاش لعميل غير مسجّل.
- لا يتطلب كل بيانات العميل — يكفي الاسم (اختياري)، وإلا يُولَّد اسم تلقائي مثل: **«عميل مؤقت رقم 1»**.
- إذا دفع العميل **كامل** قيمة الفاتورة → رصيده = صفر → يُغلق الحساب المؤقت أو يُحذف تلقائيًا (حسب إعدادات النظام).
- إذا لم يدفع الكامل → يظل الحساب المؤقت قائمًا حتى سداد المتبقي.
- عند السداد الكامل لاحقًا → يُحذف الحساب المؤقت أو يُؤرشف تلقائيًا.

**أثر على النموذج (مقترح):**
- إضافة `isTemporary Boolean @default(false)` على `Party` (مع `@map("is_temporary")`).
- إضافة حالة أرشفة: `archivedAt DateTime?` بدل الحذف الصلب (حفاظًا على القيود المرتبطة والـ audit log).
- إعداد نظام في `Settings`: **«عند تصفية الحساب المؤقت: حذف / أرشفة»** (افتراضي = أرشفة، أأمن محاسبيًا).

---

## 2) نقل الحساب المؤقت إلى عميل دائم — 🆕 جديد

**المطلوب:**
- تحويل الحساب المؤقت إلى **عميل دائم** دون فقدان أي فواتير أو معاملات سابقة.
- عند التحويل تنتقل **جميع الحركات والرصيد** إلى حساب العميل الجديد.

**أثر على النموذج (مقترح):**
- زر «تحويل لعميل دائم» في صفحة الطرف المؤقت → يفتح نموذج استكمال البيانات (هاتف/عنوان…).
- التحويل = تعديل `isTemporary → false` على نفس السجل (أبسط وأأمن: تبقى كل الروابط `invoices`/`ledgerEntries`/`treasuryTxns` كما هي بلا نقل).
- بديل (لو الدمج مع عميل قائم مطلوب): إعادة إسناد `partyId` لكل القيود داخل `$transaction` واحدة ثم **إعادة حساب سلسلة رصيد** الطرفين.

---

## 3) تحسين دورة إنشاء الفاتورة — 🟡 جزئي

**المطلوب:**
- عند اختيار عميل (مؤقت أو دائم) → **عرض الرصيد الحالي قبل حفظ الفاتورة**.  🆕
- بعد حفظ الفاتورة → تحديث رصيد العميل والخزنة تلقائيًا.  ✅ (منفّذ عبر العملية المرتبطة)
- أي قبض/سداد ناتج عن الفاتورة → يُسجَّل تلقائيًا في الخزنة.  ✅

---

## 4) التكامل بين جميع الموديولات — ✅ منفّذ (مع تحسينات مقترحة)

أي عملية في **العملاء / الموردين / الخزنة / الشيكات** تنعكس تلقائيًا على الموديولات المرتبطة، بلا تكرار تسجيل.

الحالة: منفّذ عبر «العمليات المرتبطة» (`أنشئ_عملية_مرتبطة` / `اعكس_عملية_مرتبطة` / الدفع المباشر) داخل `$transaction` ذرّية.

**تحسينات مقترحة تُلحق بهذا البند:**
- **الدفع المباشر لعميل عابر بالاسم** (بدون حساب) — منفّذ حديثًا: قيد المورد + حركة خزنة `TRANSFER` تحمل الاسم، بلا قيد على عميل.
- تحصيل قيمة الشيك عند تحوّل حالته إلى «محصّل» يُسجَّل إيرادًا في الخزنة تلقائيًا (يُراجع).

---

## 5) سجل العمليات (Audit Log) — ✅ منفّذ

- تسجيل كل عمليات الإضافة/التعديل/الحذف مع: **المستخدم، التاريخ، الوقت، نوع العملية** (+ before/after عند اللزوم).
- السجل **append-only** وقابل للمراجعة عبر `/activity-log` بفلترة بالشخص/نوع الكيان/المدى الزمني.

الحالة: منفّذ (`تسجيل_عملية` + `ActivityLog` + جدولة createdBy/updatedBy على كل سجل عمل).

---

## 6) منع الأخطاء المحاسبية — 🟡 جزئي

**المطلوب:**
- عدم السماح بحذف معاملة مرتبطة بقيود أخرى إلا بعد معالجة الارتباطات.  🟡
- عند تعديل/حذف أي فاتورة → تحديث أرصدة العميل والخزنة والشيكات (إن وُجدت) تلقائيًا.  ✅
- التأكد من عدم اختلاف أي رصيد بعد أي تعديل.  ✅ (سياسة reverse-then-reapply + `recompute()`)

**ملاحظة على البند الأول:** السياسة الحالية = **عكس ثم إعادة تطبيق** لكل الأطراف المرتبطة تلقائيًا داخل معاملة واحدة (بدل منع الحذف). المقترح: إضافة **حوار تأكيد يعرض الارتباطات** التي ستُعكَس قبل تنفيذ الحذف، لتحقيق «معالجة الارتباطات» بشكل مرئي للمستخدم.

---

## خلاصة الحالة

| البند | الحالة |
|------|--------|
| 1. الحساب المؤقت | 🆕 جديد |
| 2. تحويل مؤقت → دائم | 🆕 جديد |
| 3. دورة الفاتورة (عرض الرصيد قبل الحفظ) | 🟡 جزئي |
| 4. تكامل الموديولات | ✅ منفّذ |
| 5. سجل العمليات | ✅ منفّذ |
| 6. منع الأخطاء المحاسبية | 🟡 جزئي |

**الأولوية المقترحة للبناء:** (1) الحساب المؤقت → (2) تحويله لدائم → (3) عرض الرصيد قبل الحفظ → (6) حوار تأكيد الارتباطات.
